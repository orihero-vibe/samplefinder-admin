# Staging Environment — Design Spec

- **Date:** 2026-07-03
- **Scope:** SampleFinder workspace — admin dashboard + mobile app + shared Appwrite backend
- **Status:** Design approved; spec pending user review before planning
- **Owning repo for this doc:** `samplefinder-admin` (owns the shared Appwrite config and existing staging tooling)

## 1. Goal & Definition of Done

Stand up a complete, isolated **staging** environment so the team can QA changes end-to-end before promoting to production, while prod is in maintenance.

Done when:

1. The staging Appwrite project (`6a0ad92e0001d5e515ce`) mirrors prod: all 15 tables (attributes/indexes/permissions), the `files` bucket, all 3 functions (deployed + secrets set), auth methods, push provider, and reference data.
2. The **admin dashboard** runs and builds against staging (`npm run dev:staging` / `build:staging`).
3. A **separate-identity mobile app** ("SampleFinder (Staging)", distinct bundle/package id) runs against staging on both **Android** (locally-built APK, shared directly) and **iOS** (TestFlight), with **push notifications working**.

## 3. Key Insight — only the project ID differs

Every Appwrite resource in `appwrite/appwrite.config.json` is declared with an explicit `$id` (DB `69217af…`, bucket `6921b4ae…`, all 3 function ids, slug table ids). Appwrite namespaces IDs per project, so `appwrite push` into the staging project reproduces **identical IDs**. Therefore prod→staging client config differs **only** in `PROJECT_ID` (`691d4a54…` → `6a0ad92e…`); endpoint, DB id, collection slugs, bucket id, and function ids are unchanged. Client env files are near copy-paste; the real work is verifying backend completeness and building the side-by-side mobile app.

## 4. Decisions (approved)

- **Scope:** full — backend + admin + mobile (both platforms).
- **Mobile identity:** separate app, side-by-side (distinct bundle id, name suffix).
- **Push in staging:** yes — add staging app ids to the **existing** Firebase project (free), reuse the existing iOS APNs key.
- **iOS distribution:** TestFlight (free) over ad-hoc UDIDs.
- **Admin staging hosting:** local `dev:staging` now; a Vercel "staging" deploy is optional/later.
- **Staging deep links:** custom scheme only (`samplefinderstaging://`); no universal links (no paid domain, no conflict with the prod app).
- **Android staging signing:** reuse the existing `release.keystore` (internal test APKs).
- **Versioning:** staging shares prod version/build numbers (no independent bumping).
- **No new recurring cost:** second Appwrite project draws from the same plan quota; separate Android package is free (sideloaded APK); iOS bundle id is free within the existing Apple Developer Program; a second Firebase app in the existing project is free.

## 5. Phase 1 — Backend (verify + complete staging Appwrite)

Executed via `appwrite/staging-cli.sh` + the Appwrite console.

1. **Schema parity** — push the 15 tables + attributes + indexes + permissions; verify none missing (staging predates `popups`, `popup_interactions`, `checkins`).
2. **Bucket** — confirm/create `files` (`6921b4ae…`) with matching permissions and file/size settings.
3. **Functions** — deploy all 3; set env vars in the staging console (the `APPWRITE_FUNCTION_*` trio is auto-injected by the runtime — do **not** set those):
   - **Mobile API** — `APPWRITE_API_KEY` (staging server key), `APPWRITE_NOTIFICATION_FUNCTION_ID` (= `695d55bb…`), `TRIVIA_ONLY_TUESDAY_ET`.
   - **Notification** (schedule `*/15 * * * *`) — `APPWRITE_API_KEY`, `APPWRITE_ENDPOINT` (staging URL), `APPWRITE_PROJECT_ID` (= `6a0ad92e…`), `NOTIFICATION_APPEND_DEADLINE_MS`, `TZ`.
   - **Statistics** — `APPWRITE_API_KEY`.
   - Verify each function's scopes match the config after deploy.
4. **Push provider** — configure an **FCM Messaging provider** in staging using the same Firebase project's service account (the Notification function delivers through Appwrite Messaging).
5. **Auth** — push `settings.auth` (methods: jwt/phone/invites/anonymous/email-otp/magic-url/email-password; 1-year sessions). Configure **SMTP** (email OTP + password reset) and an **SMS provider** or `mockNumbers` (phone auth) in the staging console.
6. **Reference data** — run `clone-reference-data-to-staging.mjs` (categories/locations/tiers/settings).
7. **Seed** — create a couple of test events + a test user via the admin dashboard for QA.

## 7. Phase 3 — Mobile app

Convert static `app.json` → **`app.config.js`** driven by `APP_VARIANT` (`production` default | `staging`):

| Field | prod | staging |
|---|---|---|
| `name` | SampleFinder | SampleFinder (Staging) |
| bundle id / package | `com.samplefinder.app` | `com.samplefinder.app.staging` |
| google-services files | existing | `GoogleService-Info.staging.plist` / `google-services.staging.json` |
| deep links | `applinks:samplefinder.com` | custom scheme `samplefinderstaging://`, no universal links |
| Appwrite env | `.env` | `.env.staging` |

- **Env selection** — `babel.config.js` chooses `.env.staging` when `APP_VARIANT=staging`, else `.env`; keep the babel cache keyed on the active env file. `.env.staging` = prod `.env` with `APPWRITE_PROJECT_ID=6a0ad92e…`.
- **Deep-link plugin** — make `app.deeplink-plugin.js` variant-aware: skip Associated Domains / autoVerify App Links on staging (custom scheme only) so staging never competes with the prod app for `samplefinder.com` links.
- **Firebase** — add both staging app ids to the existing Firebase project; download the `*.staging.*` config files (referenced by `app.config.js`).
- **Build/distribute** — `APP_VARIANT=staging expo prebuild --clean` → local `gradlew` release APK (share directly) / `xcodebuild` archive → TestFlight (requires the staging bundle id registered + an App Store Connect app record).
- **Runbook** — codify the staging build sequence (extend the `release` skill) so per-variant prebuild + signing reapplication is repeatable.

- Appwrite console: set function secrets, Messaging FCM provider, SMTP/SMS.
- Apple: register `com.samplefinder.app.staging` + create the App Store Connect record for TestFlight.

## 9. Risks & must-verify items

- **`appwrite push` target** — `appwrite.config.json` hardcodes the prod `projectId`. Must verify that `staging-cli.sh` (setting `APPWRITE_PROJECT`) actually redirects `push` to staging; if the CLI reads `projectId` from the config instead, we override it per-push or temporarily swap it. **Verify before any push.**
- **Mobile prebuild/signing fragility** — `prebuild --clean` regenerates the gitignored native dirs and wipes manually reapplied signing/manifest fixes (known release fragility). Prod and staging can't coexist as native dirs — prebuild for whichever you're building. The runbook must make this explicit.
- **Function secrets** — functions won't work in staging until `APPWRITE_API_KEY` (and the vars in §5.3) are set in the console.
- **Push depends on** both the Firebase staging apps existing **and** the staging Appwrite Messaging FCM provider being configured.
- **Auth flows** (OTP, password reset, phone) won't work until SMTP/SMS (or `mockNumbers`) are configured in staging.
- **Never** put a server API key in the mobile client — staging keys stay server-side / in shell env only.

## 10. Out of scope (YAGNI)

- Vercel staging deployment of the admin (optional, later).
- A separate staging Firebase *project* (reuse the existing one).
- Staging universal-link domain / `.well-known` hosting.
- Independent staging version/build numbering.
- Play Store / public distribution of the staging app (direct APK + TestFlight only).
- Automated CI/CD for staging.

## Phasing note

One coherent spec, executed in order: **Phase 1 (backend) → Phase 2 (admin) → Phase 3 (mobile)**, because the clients are only meaningful once the backend is verified complete. The mobile phase is substantial and will get its own implementation plan in `samplefinder-app` when we reach it.
